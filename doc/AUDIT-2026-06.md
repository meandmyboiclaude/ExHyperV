### [medium] src/Services/HyperVDDAService.cs:477-481 — DDACommands blocks on async WMI query via .GetAwaiter().GetResult() on the UI thread _(compile-risk, needs decision)_

DDACommands() is invoked synchronously at line 221 inside ExecuteDdaOperationAsync, whose synchronous prefix runs on the caller's thread. The caller (DDAPageViewModel line 97) awaits ExecuteDdaOperationAsync directly from a command handler, so DDACommands executes on the WPF UI/dispatcher thread. At line 481 it calls WmiApi.QueryAsync(...).GetAwaiter().GetResult(), synchronously blocking the UI thread for the duration of a WMI query (which can take seconds). ### [medium] src/Services/VmMmioService.cs:89-117 — MMIO probe leaves VM with bogus over-limit MMIO config if final write fails or probe throws _(needs decision)_

QueryHostMmioCeilingMbAsync deliberately writes an over-limit probe MMIO config (HighMmioGapBase=2^32-1024 MB, HighMmioGapSize=1024) to the VM via ModifySystemSettings, then tries to start the VM expecting failure. It relies entirely on the caller ConfigureMmioAsync subsequently overwriting that probe value with the final config. But there is no try/finally guaranteeing restoration: if ConfigureMmioAsync's final WithObjectAsync (line 64) fails (returns false) or if anything between the probe-write and the final-write throws (caught by the outer catch at line 77 which just returns false), the VM is left persisted with the unusable over-limit MMIO probe values, which will prevent the VM from ever booting until manually fixed. The 'unexpected start success' branch (line 108) also calls StopVmAsync (force-off) without waiting for the VM to actually reach Off before the caller proceeds to ModifySystemSettings, which can fail on a still-stopping VM.

**Suggested fix:** Guard the boot-breaking probe value so it is never left persisted. The cleanest minimal change is in ConfigureMmioAsync: track whether the probe was written and, if the final write fails or an exception occurs, restore a safe baseline before returning. Because the ceiling is parsed from the start-error MESSAGE (not from the persisted value), restoring the probe to a safe baseline at any point after the start attempt is harmless.

Concretely, refactor so the final-config writer is reusable, and in ConfigureMmioAsync wrap the post-probe work in try/catch that restores a fallback config on failure:

```csharp
public async Task<bool> ConfigureMmioAsync(string vmName)
{
    bool probeWritten = false;
    try
    {
        ulong ceilingMb = await QueryHostMmioCeilingMbAsync(vmName, () => probeWritten = true);
        if (ceilingMb == 0) ceilingMb = FallbackCeilingMb;

        var resp = await WriteMmioConfigAsync(vmName, ceilingMb);
        if (resp.Success) Debug.WriteLine(Properties.Resources.MMIOOptimizer_LogConfigApplied);
        if (!resp.Success && probeWritten)
            await WriteMmioConfigAsync(vmName, FallbackCeilingMb); // restore safe baseline
        return resp.Success;
    }
    catch (Exception ex)
    {
        Debug.WriteLine(string.Format(Properties.Resources.MMIOOptimizer_LogError, ex.Message));
        if (probeWritten)
        {
            try { await WriteMmioConfigAsync(vmName, FallbackCeilingMb); } catch { /* best effort */ }
        }
        return false;
    }
}

private static async Task<ApiResponse> WriteMmioConfigAsync(string vmName, ulong ceilingMb)
{
    ulong finalBase = ceilingMb / 2;
    ulong remaining = ceilingMb - finalBase - 1024;
    ulong finalHighSize = Math.Min(remaining, 131072UL);
    ulong finalLowSize = 1024UL;
    return await WmiApi.WithObjectAsync(
        wql: RealizedSettingsWql(vmName),
        modifier: obj =>
        {
            obj["HighMmioGapBase"] = finalBase;
            obj["HighMmioGapSize"] = finalHighSize;
            obj["LowMmioGapSize"] = finalLowSize;
            obj["GuestControlledCacheTypes"] = true;
        });
}
```
(QueryHostMmioCeilingMbAsync gains an Action onProbeWritten callback invoked after setResp.Success at line 99 so the caller knows the boot-breaking value is live.)

### [medium] src/Services/VmGPUService.cs:1237 — Remote script URL/filename from index.json interpolated unescaped into sh -c command (command injection) _(needs decision)_

When deploying a non-local script, downloadCmd is built as `{proxyEnv}sh -c \"wget -q -O {remoteScriptPath} {script.SourcePathOrUrl} || curl -fL {script.SourcePathOrUrl} -o {remoteScriptPath}\"`. script.SourcePathOrUrl and script.FileName originate from a remotely-fetched index.json (ScriptBaseUrl/index.json, line 1128-1141) and are inserted unescaped into a string passed to `sh -c` and executed over SSH with the user's sudo password later in the loop. A malicious or tampered index.json could set FileName/URL containing shell metacharacters (`;`, `$(...)`, backticks, quotes) to achieve arbitrary command execution inside the VM. Additionally, proxyArg (line 1252) interpolates credentials.ProxyHost/ProxyPort unescaped into the bash command line. remoteScriptPath also embeds script.FileName (line 1217) unescaped.

**Suggested fix:** Validate FileName against a strict whitelist and reject anything non-conforming before it is ever used to build a remote path or URL. The cleanest, lowest-risk place is right after deserialization in the remote-scan loop (VmGPUService.cs ~line 1136-1142). Replace the loop body with:

    foreach (var item in remoteScripts)
    {
        // Reject tampered/malicious entries: filename must be a plain shell-safe basename ending in .sh
        if (string.IsNullOrEmpty(item.FileName) ||
            !Regex.IsMatch(item.FileName, @"^[A-Za-z0-9._-]+\.sh$"))
        {
            Debug.WriteLine($"Skipping remote script with unsafe filename: {item.FileName}");
            continue;
        }
        item.IsLocal = false;
        item.Name = string.Format(Properties.Resources.VmGPUService_LogOnline, item.Name);
        item.SourcePathOrUrl = $"{ScriptBaseUrl}{Uri.EscapeDataString(item.FileName)}";
        allScripts.Add(item);
    }

The `^[A-Za-z0-9._-]+\.sh$` whitelist guarantees FileName contains no shell metacharacters (no spaces, quotes, ;, |, $, backticks, parentheses), which makes both remoteScriptPath (line 1217) and the SourcePathOrUrl used in downloadCmd (line 1237) safe by construction. (Regex is already imported and used at line 1161.) This is preferable to escaping at line 1237 because it fixes every downstream use of FileName in one place and cannot accidentally break a legitimate query-string URL. As defense in depth, the local-scan branch already derives FileName via Path.GetFileName (line 1116) so it is not affected.

NOT auto-applied because: (1) it changes control flow (skips entries that previously would have been added) and (2) although the regex restriction is benign in practice, an injection-hardening rewrite that touches the deploy command path must be compiled and exercised on Windows, which is impossible on this Linux host. Apply manually and verify the legitimate scripts in the repo's index.json still pass the regex (they should: standard .sh filenames).

### [medium] src/Services/ProcessAffinityManager.cs:60-66, 87 — Affinity mask bit-shift (1L << i) is incorrect on systems with more than 64 logical processors _(needs decision)_

GetVmProcessAffinity loops `for (int i = 0; i < Environment.ProcessorCount; i++)` and tests `(affinityMask & (1L << i))`; SetVmProcessAffinity does `newAffinityMask |= (1L << coreId)`. In C# the shift count of a `long` is masked to mod 64, so when Environment.ProcessorCount > 64 (or coreId >= 64) the shift wraps (e.g. `1L << 64` == `1L << 0` == 1) and the wrong bits are read/written. process.ProcessorAffinity is also a single-processor-group (max 64) value on Windows, so on >64-LP hosts the computed mask is silently wrong, mapping core 64 onto core 0, etc. Failure scenario: high-core-count host (Threadripper/EPYC/dual-socket) — setting/reading vmmem affinity targets the wrong cores.

**Suggested fix:** Not a single unambiguous one-liner (3 sites, and a design choice between clamp-vs-reject-vs-full-processor-group-support), so do not auto-apply. The minimal, non-corrupting fix that compiles cleanly on net8.0-windows:

1) GetVmProcessAffinity read loop (line 60):
   `for (int i = 0; i < Math.Min(Environment.ProcessorCount, 64); i++)`

2) SetVmProcessAffinity setter loop (line 87): guard against out-of-range coreIds so they cannot alias onto low bits:
   `foreach (int coreId in coreIds) if (coreId >= 0 && coreId < 64) newAffinityMask |= (1L << coreId);`

3) SetVmProcessAffinity "use all cores" fallback (line 92) — fix the shift-by-ProcessorCount which breaks at exactly 64 LPs:
   `int bits = Math.Min(Environment.ProcessorCount, 64);`
   `process.ProcessorAffinity = (IntPtr)(bits >= 64 ? -1L : ((1L << bits) - 1));`
   (-1L sets all 64 bits; for <64 LPs `(1L << bits) - 1` yields the correct low-bit mask.)

All edits use only Math.Min and standard long/IntPtr arithmetic already in scope — no new usings, no API changes, no compile risk. NOTE: this only prevents silent bit-aliasing within processor group 0; addressing cores beyond 64 (multiple processor groups) requires SetThreadGroupAffinity/GetProcessGroupAffinity P/Invoke and is a separate, larger change worth a follow-up issue.

### [medium] src/Linux/script/CachyOS-Official.sh:432-444 (rcfiles + usermod), and Fedora-44-Official.sh 551-562 — Per-user steps (~/.bashrc/.zshrc edits and usermod $USER) run as root, targeting the wrong user/home _(needs decision)_

The whole script is launched as root (VmGPUService.cs:1255 wraps it in `sudo -S -E bash <script> deploy ...`). Inside, the graphics block edits ~/.bashrc / ~/.zshrc and runs `sudo usermod -a -G video,render $USER`. When EUID is 0, ~ expands via $HOME which is /root on distros with sudo's default always_set_home (Fedora always sets HOME=/root), so the GPU-PV export lines are written to /root's rc files and never affect the interactive login user's shell. Likewise `$USER` under sudo is commonly 'root' (or absent), so usermod either adds root to video/render (pointless) or errors, instead of granting the real desktop user GPU access. Net effect: env vars and group membership silently apply to root, and the actual user still cannot use the virtual GPU until they fix it manually.
